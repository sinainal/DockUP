## 1. `POST /api/results/scan` uses cached subfolder as implicit default

- File: `docking_app/routes/results.py:23`
- Current behavior: when `root_path` is omitted, the endpoint falls back to `STATE["results_root_path"]` before `"data/dock"`.
- Observed effect: an empty request can scan a previously selected nested dock folder instead of the actual dock root.
- Reproduction:
  - Start app.
  - Change results root in UI to a nested dock folder.
  - Call `POST /api/results/scan` with `{}`.
  - Response `root_path` may be that cached nested folder, not `data/dock`.
- Risk:
  - surprising stateful API behavior
  - inconsistent UI after reload
  - tests or scripts expecting root-level scan can get partial data

## 2. Resume flow stores `out_root_path` in a different format

- File: `docking_app/routes/core.py:915`
- Current behavior:
  - fresh queue build stores `out_root_path` as display-relative path like `data/dock/...`
  - resume preparation stores `out_root_path` as absolute filesystem parent path
- Risk:
  - UI receives mixed path formats for the same field
  - path resolution and state persistence become harder to reason about
  - reload/resume regressions are likely

## 3. Fresh-machine template rendering failure (`TypeError: unhashable type: 'dict'`)

- Screenshot symptom:
  - stack ends in `jinja2/environment.py -> _load_template -> cache.get(cache_key)`
  - final error: `TypeError: unhashable type: 'dict'`
- Relevant template calls:
  - `docking_app/routes/core.py:297`
  - `docking_app/ligand_3d/app.py:559`
- Current code uses positional arguments:
  - `templates.TemplateResponse(request, "index.html", {"title": ...})`

### Root-cause analysis

Most likely cause is dependency-version drift on a fresh machine.

- The project uses wide-open dependency ranges:
  - `fastapi>=0.110.0`
  - `uvicorn[standard]>=0.29.0`
  - `jinja2>=3.1`
  - and similar in `pyproject.toml` and `requirements/core.txt`
- `setup.sh` installs from those open ranges, so a clean install can resolve to a different FastAPI/Starlette/httpx stack than an already-working machine.
- I verified this environment is currently on:
  - `fastapi 0.128.0`
  - `starlette 0.50.0`
  - and both `/` routes render correctly here.
- I also tested fresh isolated virtualenvs with other version combinations and immediately hit compatibility instability in the stack itself, for example:
  - old Starlette test client requiring extra `httpx`
  - older Starlette/FastAPI plus newer `httpx` breaking `TestClient` construction with `Client.__init__() got an unexpected keyword argument 'app'`

This makes the screenshot plausible even if it does not reproduce in the current environment:

- fresh install resolves a different Starlette/FastAPI/Jinja interaction path
- positional `TemplateResponse(...)` usage is more fragile across version changes
- the framework ends up passing the wrong object as template name/cache key, which matches the observed `dict` reaching Jinja template cache lookup

### Strong suspicion

Even if the exact failing version tuple is still unknown, the problem class is clear:

- unpinned framework dependencies
- positional `TemplateResponse(...)` calls that rely on framework argument interpretation across versions

### Recommended fix direction

1. Make template calls version-stable by using keyword arguments:
   - `TemplateResponse(request=request, name="index.html", context={...})`
2. Pin or constrain the FastAPI/Starlette/httpx stack to a known-good set.
3. Add a clean-environment smoke test that only boots the app and requests `/`.

### Docker note

- No `Dockerfile`, `docker-compose.yml`, or `.dockerignore` currently exists in the repo root, so I could not run an actual Docker-based reproduction from repository assets.
- I used isolated virtualenvs instead to approximate a zero-machine install.
